The key insight

A CLAUDE.md alone tells AI what your project is. A domain handbook (like d3handbook.md) tells AI how to write code in it — every pattern, helper API, convention, and decision. Combined with Claude Skills, developers get one-command workflows that produce code matching your codebase on the first try.

Layer 1

CLAUDE.md

Short project overview. Points to the handbook. Auto-loaded every session.

Layer 2

Domain Handbook

Deep context: data models, component patterns, helper APIs, conventions, Tableau mapping. The real differentiator.

Layer 3

Claude Skills

Reusable prompt templates: /new-chart, /migrate-tableau, /review-chart. One command, full workflow.

Why a Handbook, not just CLAUDE.md?

Most guides stop at CLAUDE.md. That's like giving a new hire a one-page onboarding doc. Here's the difference:

Project Structure

Place these files at your project root. Claude Code auto-discovers CLAUDE.md and .claude/.

CLAUDE.md — The Pointer

Keep it short. Its job is to orient AI and point to the handbook for depth.

d3handbook.md — The Deep Context

This is the real differentiator. It captures everything a developer (human or AI) needs to write code that fits your codebase: data model architecture, component patterns, helper API signatures, convention rules, and a Tableau→D3 mapping table. See d3handbook.md at the project root for the full file.

Full file — ready to drop into your project root
What makes a good handbook?
  • - Interface definitions with field-level comments
  • - Exact function signatures and their behavior
  • - The component pattern as a code template (not just description)
  • - Convention rules with the why (e.g. "highlight not hide — because users need to see the dimmed context")
  • - Step-by-step checklists for common tasks
  • - Mapping tables for domain concepts (Tableau→D3 in our case)

Claude Skills

Skills are reusable prompt templates stored in .claude/skills/<name>/SKILL.md. Each becomes a slash command. They combine your handbook context with a structured workflow.

SKILL.md Format

Every skill has YAML frontmatter (configuration) + markdown body (instructions).

Key frontmatter options

name
Becomes the /slash-command name
description
When Claude should auto-invoke this skill
argument-hint
Shown in autocomplete, e.g. [chart-name]
disable-model-invocation
Only manual invoke (prevents auto-trigger)
context: fork
Run in isolated subagent (for reviews/research)
agent: Explore
Read-only subagent type (safe for audits)

Variable substitution

$ARGUMENTS
Everything after the command name
$0, $1, $2
Positional arguments (space-separated)
!`shell cmd`
Shell output injected before Claude sees it
{{'$\{CLAUDE_SKILL_DIR\}'}}
Path to the skill's directory

Skill: /new-chart

Scaffolds a complete chart component. Usage: /new-chart scatter

manual only accepts arguments

Skill: /migrate-tableau

Converts a Tableau worksheet description into D3 code following your patterns.

manual only interactive

Skill: /review-chart

Audits a chart component against your conventions. Usage: /review-chart src/app/charts/d3-bar/d3-bar.component.ts

runs in isolated fork read-only (Explore agent)

Pre-approve safe commands in .claude/settings.json so AI doesn't ask for confirmation every time.
